**SOLID 是面向对象设计的五大核心原则**，由 Robert C. Martin（Bob 大叔）提出，用于指导写出**高内聚、低耦合、易维护、易扩展**的代码。

### 一、SOLID 原则总览




| 字母  |        原则全称        |             核心思想             |
| :---: | :--------------------: | :------------------------------: |
| **S** | **单一职责原则 (SRP)** |   一个类 / 模块只负责一项职责    |
| **O** | **开放封闭原则 (OCP)** |      对扩展开放，对修改封闭      |
| **L** | **里氏替换原则 (LSP)** | 子类可完全替换父类，不破坏正确性 |
| **I** | **接口隔离原则 (ISP)** |     客户端不依赖不需要的接口     |
| **D** | **依赖倒置原则 (DIP)** |      依赖抽象，而非具体实现      |

------

### 二、各原则详解（含通俗理解）

#### 1. S - 单一职责原则 (Single Responsibility Principle)

**核心**：一个类应该只有一个引起它变化的原因。

- 通俗：**一个类只干一件事**。
- 反例：一个类同时做 “用户注册、发邮件、写日志、存数据库”。
- 正例：拆成 `UserService`、`EmailService`、`Logger`、`UserRepository` 各司其职。

#### 2. O - 开放封闭原则 (Open-Closed Principle)

**核心**：软件实体（类、模块、函数）**对扩展开放，对修改封闭**。

- 通俗：**加新功能靠新增代码，不改老代码**。
- 做法：用**抽象、接口、多态、装饰器、策略模式**等实现扩展。
- 好处：老代码稳定、少改少 Bug、易回归测试。

#### 3. L - 里氏替换原则 (Liskov Substitution Principle)

**核心**：子类必须能**完全替换**父类，且程序行为不变。

- 通俗：**子类不能破坏父类的契约**。
- 反例：父类 “鸟能飞”，子类 “鸵鸟” 继承鸟但不能飞 → 违反 LSP。
- 正例：设计 `Flyable` 接口，只有会飞的鸟实现它。

#### 4. I - 接口隔离原则 (Interface Segregation Principle)

**核心**：客户端不应该被迫依赖它**不需要**的接口。

- 通俗：**接口要小而专，不要大而全**。
- 反例：一个 `IAnimal` 包含 `eat()`、`fly()`、`swim()` → 鱼实现时被迫空实现 `fly()`。
- 正例：拆成 `IEatable`、`IFlyable`、`ISwimable`，按需实现。

#### 5. D - 依赖倒置原则 (Dependency Inversion Principle)

**核心**：

1. 高层模块不依赖低层模块，**都依赖抽象**。
2. 抽象不依赖细节，**细节依赖抽象**。

- 通俗：**依赖接口 / 抽象类，不依赖具体类**。
- 做法：用**依赖注入（DI）**、面向接口编程。
- 好处：解耦、易替换实现、易单元测试。

------

### 三、一句话总结

**SOLID = 职责单一 + 开闭扩展 + 子类可替换 + 接口要小 + 依赖抽象**。

遵循它能让代码更干净、更易维护、更能应对需求变化。
